Pattern


USE CASES
  • With the help of pattern keys, which can only be recognised by users who have registered accounts, this project manages security attacks made through social engineering.
  • The Fail Success [f & s] input key is handled by the pattern key when registering an account. Every time you connect into your account, this key is used.
  • The distinction between authorised users and anonymous users is that authorised users must log in on their first attempt and then check their pattern key to see whether their account matches. If it does, your website will then reroute to the admin page. Whereas in this instance, since Anonymous person logged in on their first attempt without understanding the concept of a pattern key, it is necessary for them to wait for admin permission or rejection.
  • By employing this pattern key technique, even if attackers gain access to our account and have our user ID and secret key, they will be unable to log in without first knowing the pattern key layer. This layer protects the attacker by confusing what occurs in the backend and preventing them from accessing the data until admin approval.
  • One might wonder what would happen if the compromised data contained information about a pattern key. To prevent this personal information leakage, the pattern key details will be configured into numerous different servers, making it impossible for an attacker to determine which IP address the pattern key has been saved on.
  • The existing services like Oauth, OpenID, SAML, JWT, Kerberos, etc. are overcome by providing this pattern key layer.
